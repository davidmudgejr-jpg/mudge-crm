# Phase 1 — Static Code Audit

## 1.1 Frontend Pages (16 pages)

### [MAJOR] DedupReview bypasses the global 401 refresh interceptor
- **File:** `ie-crm/src/pages/DedupReview.jsx:58-62, 78, 127`
- **Symptom:** `DedupReview` constructs its own `headers` object and calls raw `fetch()` instead of `bridge.db.*`. The global 401 refresh-and-retry lives inside the fetch interceptor installed by `AuthContext` — DedupReview never goes through it, so when the token expires mid-session the page shows a silent toast error and clusters stay empty. A hard refresh is needed to re-auth.
- **Root cause:** DedupReview predates the AuthContext refresh mechanism and was never migrated.
- **Impact:** Every user hitting the dedup review flow for >14 days without a browser refresh will see a dead page instead of fresh data.
- **Severity:** MAJOR · **Effort:** MEDIUM
- **Recommendation:** Add a `bridge.dedup = { clusters, merge, reject, ... }` namespace that routes through the same patched `fetch` bridge uses, then rewrite DedupReview to use it. That's ~60 lines of new code in `bridge.js` and ~5 replacement points in DedupReview.

### [MAJOR] Import page — file parsing errors not surfaced
- **File:** `ie-crm/src/pages/Import.jsx`
- **Symptom:** The import flow chains `bridge.file.parse()` without a `.catch()` on every branch. If a corrupt PDF/Excel throws mid-parse, the import preview modal never populates and the user sees a greyed-out "Continue" button with no error message.
- **Severity:** MAJOR · **Effort:** MEDIUM
- **Recommendation:** Wrap each file-op call in try/catch and route rejected promises into the existing `addToast('error', …)` path. Test with a deliberately corrupted `.xlsx`.

### [MINOR] Detail pages partial data on linked-query failure
- **Files:** `ContactDetail.jsx`, `CompanyDetail.jsx`, `PropertyDetail.jsx`, `DealDetail.jsx`
- **Symptom:** Uses `Promise.allSettled` for M2M fetches (good) but doesn't distinguish partially-successful loads from fully-successful. Non-blocking, defaults to empty arrays.
- **Severity:** MINOR · **Effort:** EASY · **Recommendation:** No action needed; current behavior is correct.

### [MINOR] 33 pages emit `console.error` with no telemetry backend
- **Symptom:** Production errors only go to the browser console; no Sentry / LogRocket integration.
- **Severity:** MINOR · **Effort:** EASY · **Recommendation:** Defer until post-MVP; add Sentry in a focused pass.

### What looks solid
- Login.jsx, Contacts, Companies, Properties, ActionItems, TPE, TPEEnrichment — all use try/catch, render loading/empty/error branches, and validate inputs before submit.
- `useFetchGuard` prevents stale-state writes when components unmount mid-request.
- Every page uses `useAuth()` and destructures only `{ user, login, logout }` (safe fields) except DedupReview (above).

## 1.2 Frontend API layer + AuthContext

### [CRITICAL] bridge.js caches failed promises for 30s → transient errors become sticky
- **File:** `ie-crm/src/api/bridge.js:60-84`
- **Symptom:** The GET cache stores `{ promise, expiry }`. If a request throws (network blip, 5xx), the rejected promise stays in the cache until TTL expires. Every subsequent call for 30 seconds `.catch()`es the same rejection, returning the old failure instead of retrying. Users see "Failed to load" on a recovered network for half a minute.
- **Root cause:** No `.catch()` on the cache-write path.
- **Severity:** CRITICAL · **Effort:** MEDIUM
- **Recommendation:**
  ```js
  const promise = (async () => {
    try { /* fetch */ }
    catch (e) { GET_CACHE.delete(path); throw e; }
  })();
  GET_CACHE.set(path, { promise, expiry: Date.now() + CACHE_TTL });
  ```

### [MAJOR] Cache invalidation is too coarse — every `/api/db/*` write wipes all entity caches
- **File:** `ie-crm/src/api/bridge.js:34-56`
- **Symptom:** `invalidateCache` computes `path.split('/').slice(0,3).join('/')` → always resolves to `/api/db`. A POST to `/api/db/contacts` therefore invalidates `/api/db/contacts`, `/api/db/companies`, `/api/db/deals`, etc. The 30s cache ends up useless for multi-entity interactions (e.g., during a Claude AI batch).
- **Severity:** MAJOR · **Effort:** EASY
- **Recommendation:** Parse entity from `path.split('/')[3]` and invalidate only matching entity caches. ~5 lines.

### [MAJOR] `logout()` doesn't clear bridge cache or abort in-flight requests
- **File:** `ie-crm/src/contexts/AuthContext.jsx:76-79`
- **Symptom:** `logout()` removes token + resets user state, but doesn't call `clearApiCache()` or abort pending fetches. On a shared device, user A's cached data leaks into user B's session for up to 30s.
- **Severity:** MAJOR · **Effort:** MEDIUM (cache clear EASY; AbortController adoption MEDIUM)
- **Recommendation:** Import `clearApiCache` from `bridge.js` and call it inside `logout()`. Add an `AbortController` ref that aborts on logout.

### [MAJOR] Concurrent 401s trigger premature logout via `retryingUrls` Set
- **File:** `ie-crm/src/contexts/AuthContext.jsx:161,178-184`
- **Symptom:** The Set tracks URL-level retry state, but two concurrent requests to the same URL can race: request A fails 401 → adds URL to Set → refresh starts · request B fails 401 → sees URL already in Set → logs the user out immediately ("Retry also returned 401").
- **Severity:** MAJOR · **Effort:** HARD
- **Recommendation:** Replace with `Map<url, retryCount>` and only log out after 2+ consecutive 401s per URL. Or use Promise-based deduplication: both requests await the same refresh promise.

### [MINOR] Undo logging is opt-in — only ClaudePanel calls `logUndo`
- **File:** `ie-crm/src/api/database.js:1152-1157` + `ie-crm/src/components/ClaudePanel.jsx:418`
- **Symptom:** The stated invariant "all writes are undoable" is false — only Claude-initiated writes are. UI inline edits (CrmTable, QuickAdd, link/unlink) don't log undo.
- **Severity:** MINOR · **Effort:** HARD · **Recommendation:** Either move undo logging into `db.update/create/remove` automatically, or update docs to reflect reality.

### [MINOR] JWT decode doesn't validate segment count before `atob`
- **File:** `ie-crm/src/contexts/AuthContext.jsx:37,51`
- **Symptom:** Bad tokens throw inside a try/catch that swallows silently but never clears the bad token from localStorage — the 401 loop keeps firing.
- **Severity:** MINOR · **Effort:** EASY · **Recommendation:** `if (parts.length !== 3) { localStorage.removeItem(TOKEN_KEY); return; }`

### [MINOR] views.js doesn't wrap `fetch` for network errors
- **File:** `ie-crm/src/api/views.js:6-20` — no try/catch around fetch itself, so network failures crash the caller.

### What looks solid
- SQL injection is well-defended on the client side: `validateFieldKeys`, `ALLOWED_COLS`, parameterized search.
- JWT refresh deduplication (`refreshingRef` + `refreshPromiseRef`) is correct under same-URL refresh concurrency.
- `Authorization: Bearer` + `Content-Type: application/json` consistent across all calls.

## 1.3 Backend Routes (~150 endpoints)

### [CRITICAL] JWT fallback secret still present in source
- **File:** `ie-crm/server/middleware/auth.js:14`
- **Symptom:** `const EFFECTIVE_JWT_SECRET = JWT_SECRET || 'dev-secret-DO-NOT-USE-IN-PRODUCTION';` — if the env var is ever unset in Railway (config drift, env split on preview deploys, etc.), tokens become forgeable with a string that is **in the public repo**.
- **Root cause:** The file logs a warning but only hard-exits if `NODE_ENV === 'production'`, and that check happens at require-time; if Railway ever ships without `NODE_ENV=production` set, the fallback is silently used.
- **Impact:** Full auth bypass. Any attacker can forge admin tokens.
- **Severity:** CRITICAL · **Effort:** EASY
- **Recommendation:** Remove the fallback entirely: `if (!JWT_SECRET) throw new Error('[FATAL] JWT_SECRET is required');`. Also add a startup health check `/api/health/ready` that returns 503 if `JWT_SECRET` isn't set. **Do this first.**

### [CRITICAL] `X-Agent-Key` comparison is not constant-time (timing attack)
- **File:** `ie-crm/server/routes/ai.js:68`
- **Symptom:** The agent key check uses plain `agentKey === validKey` string equality. An attacker with network access can brute-force the key byte-by-byte via timing side channels.
- **Severity:** CRITICAL · **Effort:** EASY
- **Recommendation:** Use `crypto.timingSafeEqual(Buffer.from(a), Buffer.from(b))` with a length check first.

### [MAJOR] AI command routes bypass `undo_log`
- **Files:** `ie-crm/server/routes/ai.js` — `/fill-empty` (1565-1669), `/log-interaction` (1671-1718), `/create-task` (1721-1787)
- **Symptom:** Direct writes to `contacts`, `interactions`, `action_items` via the agent command path do not insert into `undo_log`. A compromised or buggy agent can create/modify records with no reversal trail.
- **Severity:** MAJOR · **Effort:** MEDIUM
- **Recommendation:** Wrap every write inside `/api/ai/command/*` with an `undo_log` insert. Also enforce `req.authType === 'agent' && req.agentName === 'houston_command'` OR `req.user.role === 'admin'` on these routes.

### [MAJOR] DB error messages leak to clients
- **File:** `ie-crm/server/index.js:686, 726, 746, 816, 1890` (5+ instances)
- **Symptom:** `/api/db/update`, `/api/db/link`, `/api/db/unlink`, `/api/db/create`, `/api/import/batch` all `res.status(500).json({ error: err.message })` — which leaks Postgres error text including **column names, table names, and constraint names**. Useful to attackers for schema fingerprinting.
- **Severity:** MAJOR · **Effort:** EASY
- **Recommendation:** Wrap with `const msg = process.env.NODE_ENV === 'production' ? 'Operation failed' : err.message;` and log the real error to stderr only.

### [MAJOR] `/api/claude/chat` has no write-rate-limiter
- **File:** `ie-crm/server/index.js:893-962`
- **Symptom:** Only the global `apiLimiter` (200/min) applies. A malicious prompt or prompt-injection in a CSV filename could trigger the AI to auto-execute dozens of writes in one request.
- **Severity:** MAJOR · **Effort:** EASY
- **Recommendation:** Apply `aiLimiter` (30/min) as an additional gate and cap `messages.length` + total characters before forwarding to Claude.

### [MAJOR] Verification routes dual-auth lacks admin gating on `/resolve`
- **File:** `ie-crm/server/routes/verification.js`
- **Symptom:** Agents can create verification requests (good) but any agent holding `X-Agent-Key` can also resolve/approve them (bad). The resolution path should be admin-only.
- **Severity:** MAJOR · **Effort:** MEDIUM
- **Recommendation:** `if (req.authType === 'jwt' && req.user.role !== 'admin') return res.status(403)`. Agents can request, only humans can resolve.

### [MINOR] Bulk-delete is admin-gated but doesn't log per-row audit
- **File:** `ie-crm/server/index.js:1287-1308` — parameterized & role-gated (good), but no `undo_log` insert per deleted row.
- **Recommendation:** `DELETE ... RETURNING *` then loop into `undo_log`.

### [MINOR] `/api/db/read` relies on implicit `requireAuth` via global mount
- **File:** `server/index.js:314` — `app.use('/api', requireAuth)`. Correct today but fragile — adding a new route before line 314 would bypass auth. Make the check explicit per-route.

### [MINOR] Contracts `formCode` not whitelisted
- **File:** `server/routes/contracts.js:66-68` — `path.join(PARSED_DIR, formCode + '.json')`. Low practical risk (PARSED_DIR is hardened) but easy to harden further: `if (!/^[A-Z0-9_-]+$/.test(formCode)) return 400`.

### [MINOR] Knowledge graph visibility hardcoded to emails
- **File:** `server/routes/knowledge.js:63-64` — should use `req.user.role === 'admin'` instead.

### What looks solid
- 221 try/catch blocks across 4,181 lines (≈1 per 19 lines).
- Parameterized SQL throughout — no string concatenation of user input.
- Dedup merge path uses `BEGIN/COMMIT/ROLLBACK` with per-row SAVEPOINTs.
- Date type override (lines 133-140) keeps `DATE/TIMESTAMP` as strings and avoids TZ shift bugs.
- Rate limiters are configured (`authLimiter` 10/15min, `apiLimiter` 200/min, `aiLimiter` 30/min).

## 1.4 Backend Services + Middleware

### [CRITICAL] `deleteFile()` accepts arbitrary URLs → path-traversal file delete
- **File:** `ie-crm/server/services/fileUpload.js:73-76`
- **Symptom:** `path.join(UPLOADS_DIR, '..', url)` — if a caller passes `/uploads/../../.env` as the URL, the function resolves to the repo's `.env` and deletes it. If the URL comes from a DB row controlled by an agent or upload caller, this is a real attack vector.
- **Severity:** CRITICAL · **Effort:** MEDIUM
- **Recommendation:** Extract only the basename from the URL, reject any path containing `..`/`/` / null bytes. Better: store file UUIDs in the DB and reconstruct paths server-side from `(folder, uuid)` pairs.

### [MAJOR] Orphaned uploads on DB-insert failure
- **File:** `ie-crm/server/services/fileUpload.js:17-52` + `server/index.js:3064-3071`
- **Symptom:** Upload hits Blob/disk first, then DB insert. If the insert fails (validation, duplicate, transient error), the file is left on Blob forever. No cron cleanup.
- **Severity:** MAJOR · **Effort:** MEDIUM
- **Recommendation:** Wrap in a try/finally that deletes on failure, or insert a pending DB row first and flip a flag after upload succeeds.

### [MAJOR] SQL column whitelist is case-sensitive
- **File:** `ie-crm/server/utils/sqlSafety.js:18-125`
- **Symptom:** `allowed.has(col)` — Postgres unquoted identifiers are case-folded to lowercase, so `Email_1` is logically valid but rejected by validation. Real incoming data from user edits or imports can fail here.
- **Severity:** MAJOR · **Effort:** EASY
- **Recommendation:** Normalize: `allowed.has(col.toLowerCase())` and pass the normalized form to the query builder.

### [MAJOR] Instantly service has no retry / no Retry-After parsing
- **File:** `ie-crm/server/services/instantly.js:40-49`
- **Symptom:** 429 / 5xx responses are surfaced as generic errors with no exponential backoff and no `Retry-After` header parsing. Rate-limit bursts poison outbound campaigns.
- **Severity:** MAJOR · **Effort:** MEDIUM
- **Recommendation:** Add bounded-retry (3 attempts, exponential backoff, honoring `Retry-After`), distinguish retryable vs. permanent failures.

### [MAJOR] Missing `INSTANTLY_API_KEY` throws instead of returning a clean 503
- **File:** `ie-crm/server/services/instantly.js:13-17` — unhandled throw propagates up as an unhandled promise rejection.
- **Recommendation:** Graceful-degrade: if the key is missing, return 503 with `{ error: 'Instantly integration not configured' }` and disable the feature.

### [MINOR] No SQL `ORDER BY` direction validator
- **File:** `ie-crm/server/utils/sqlSafety.js` — there's a column whitelist but no `validateDirection(dir)` helper. Each route hand-rolls its own check.
- **Recommendation:** Add one helper: `const validateDir = (d) => (['asc','desc'].includes((d||'').toLowerCase()) ? d.toLowerCase() : 'asc')`.

### [MINOR] Address normalizer doesn't special-case P.O. boxes; composite matcher thresholds hardcoded; Instantly error logs truncated
- All 3 are quality-of-life issues that don't affect production today.

### What looks solid
- JWT signature verification via `jwt.verify` (full validation of sig, exp, format).
- File MIME whitelist + 10 MB size cap in multer config.
- Per-IP + per-agent-key rate limiting.
- Temp-file cleanup after successful Blob upload.
- The address normalizer's Levenshtein implementation is correct.

  2. Add a `prestart` check that errors if any migration filename contains a space.

- **Severity:** MAJOR · **Effort:** MEDIUM

This requires updating any tracked-state records, but the DB has no `schema_migrations` table yet so there's no state to update — do this migration cleanup alongside adding the tracking table.

## 3.4 Auth flows

### [CRITICAL] Auth rate limiter is **broken in production** (pre-existing bug)
- **File:** `ie-crm/server/index.js:95-114, 201` · `trust proxy` setting at line 28
- **Symptom:** Hit `/api/auth/login` 12 times rapidly with bad credentials. Every response returned `ratelimit-limit: 10 / ratelimit-remaining: 9`. The counter **never advances past 9** — every request looks like the first request of a new window. No 429 is ever emitted.
- **Root cause:** Deployed behind Fastly → Railway edge → Express. `app.set('trust proxy', 1)` strips exactly one proxy hop, but there are effectively two proxies in front (Fastly plus Railway's internal edge). The `X-Forwarded-For` chain isn't parsed deep enough to reach the real client IP, and the header on every request contains a *different* Fastly edge-cache-node IP (cache `khhr2060037-HHR` visible in `x-railway-cdn-edge` header). Since `express-rate-limit`'s default `keyGenerator` uses `req.ip`, every request ends up with a unique key → brand-new limiter state → infinite attempts.
- **Impact:** **Login brute-force is unblocked.** An attacker can run unlimited password attempts against any email in the system. The `bcryptjs` hashing helps (slow per attempt) but doesn't compensate for the loss of rate limiting. The same bug applies to `/api/auth/change-password` (also wrapped in `authLimiter` at line 278).
- **Severity:** CRITICAL · **Effort:** EASY
- **Recommendation:**
  1. Bump `app.set('trust proxy', 1)` to `app.set('trust proxy', 2)` so Express strips both proxies (Fastly + Railway) and `req.ip` becomes the real client IP.
  2. Add a custom `keyGenerator` that falls back to a stable fingerprint (e.g., the real client IP from `req.headers['cf-connecting-ip'] || req.headers['fastly-client-ip']`) if `req.ip` is still the proxy.
  3. Remove `validate: false` so `express-rate-limit` will warn about bad proxy configs in future.
  4. Consider moving to a Redis-backed store so limits survive container restarts across Railway replicas.
  5. **Post-fix: verify** with the same 12-POST curl loop and confirm `ratelimit-remaining` decrements 9→8→7→…→0→429.

### What works on auth

## 3.5 API contracts — bad-payload probes

### [MAJOR] `/api/db/create` leaks raw Postgres errors to the client (confirmed in production)
- **File:** `ie-crm/server/index.js:817` (inside the `catch` of `POST /api/db/create`)
- **Evidence captured live:**
  - Creating a contact with `type: "prospect"` → `{"error":"new row for relation \"contacts\" violates check constraint \"chk_contact_type\""}` (leaks constraint name)
  - Creating a contact with `email: "x"` → `{"error":"column \"email\" of relation \"contacts\" does not exist"}` (leaks schema information)
- **Root cause:** The `catch` handler is `res.status(400).json({ error: err.message })` with no `NODE_ENV === 'production'` guard. By contrast, `/api/db/read` at line 557 DOES have the guard and returns `{"error":"Query failed"}` in prod — so the error-masking pattern exists in the codebase, it just isn't applied uniformly.
- **Impact:** Schema fingerprinting for attackers. Every failed create reveals a constraint name, a column name, or an enum mismatch. Combined with the broken auth rate limiter, an attacker can probe the schema from the login side-channel too.
- **Severity:** MAJOR · **Effort:** EASY
- **Recommendation:** Copy the `safeMessage` pattern from line 557 into lines 686, 726, 746, **817**, and 1890. Log the real error to stderr, return a generic string to the client. A 10-line change.

### [MAJOR] `SERVER_ALLOWED_COLS.contacts` has phantom `email`, `phone`, `mobile` columns that don't exist in the DB
- **File:** `ie-crm/server/index.js:625-627` (inside `SERVER_ALLOWED_COLS`)
- **Symptom:** The whitelist for `contacts` lists `email`, `phone`, `mobile` as valid columns. The actual DB schema has `email_1`, `email_2`, `email_3`, `phone_1`, `phone_2`, `phone_3`. If anything (UI, importer, agent, Claude) tries to create a contact with a field named `email`, the whitelist passes it through → INSERT fails at the Postgres layer → user sees `column "email" of relation "contacts" does not exist` (thanks to the error-leak bug above, the frontend also gets the raw message).
- **Root cause:** The whitelist was never updated when the DB schema moved from single-field contact info to multi-field (`email_1/email_2/email_3`). This is the same class of issue the 2026-04-14 `lessons.md` entry flagged — agents writing `decision_maker_title` when the column didn't exist — and the fix pattern should be the same: **validate whitelist entries against `information_schema.columns` at boot time** and fail fast on drift.
- **Impact:** Creating a contact with email via `/api/db/create` is completely broken. The frontend uses a different path (likely `/api/db/update` with `email_1`), but any new caller — an agent, the import UI if it ever adds email-on-create, a Claude-generated workflow — will hit this wall. It's a landmine.
- **Severity:** MAJOR · **Effort:** EASY
- **Recommendation:** Replace the `contacts` whitelist with the real column list: `full_name`, `first_name`, `last_name`, `email_1`, `email_2`, `email_3`, `phone_1`, `phone_2`, `phone_3`, `title`, `notes`, `city`, `tags`, `overflow`, `type`, `data_source` (plus the rest of the real schema).
